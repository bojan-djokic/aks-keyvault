01 — Overview

System Architecture

AKS + Azure Key Vault integration allows Kubernetes pods to securely access secrets, keys, and certificates without ever embedding credentials in code or container images. The architecture leverages Microsoft Entra Workload Identity for passwordless authentication.

02 — Integration Patterns

Integration Methods

Three primary patterns for connecting AKS workloads to Azure Key Vault, each with distinct trade-offs in security, complexity, and operational overhead.

Secrets Store CSI Driver Recommended

The Azure Key Vault Provider for Secrets Store CSI Driver is the Microsoft-recommended approach. It mounts secrets as volumes or syncs them to Kubernetes secrets, with support for automatic rotation.

✅
Works with Workload Identity (zero credentials) or User-Assigned Managed Identity. Supports auto-rotation, Kubernetes secret sync, and version pinning.
SecretProviderClass — YAML 
apiVersion: secrets-store.csi.x-k8s.io/v1
kind: SecretProviderClass
metadata:
  name: azure-kvname-workload-identity
  namespace: my-app
spec:
  provider: azure
  secretObjects:                   # Sync to K8s Secret
    - secretName: foosecret
      type: Opaque
      data:
        - objectName: secret1
          key: SECRET_KEY
  parameters:
    usePodIdentity: "false"
    clientID: "<USER_ASSIGNED_CLIENT_ID>"
    keyvaultName: "my-key-vault"
    cloudName: ""
    objects: |
      array:
        - |
          objectName: secret1
          objectType: secret
          objectVersion: ""
        - |
          objectName: mycert
          objectType: cert
    tenantId: "<TENANT_ID>"
Pod Spec with Volume Mount 
spec:
  serviceAccountName: workload-identity-sa
  volumes:
    - name: secrets-store01-inline
      csi:
        driver: secrets-store.csi.k8s.io
        readOnly: true
        volumeAttributes:
          secretProviderClass: "azure-kvname-workload-identity"
  containers:
    - name: my-app
      volumeMounts:
        - name: secrets-store01-inline
          mountPath: "/mnt/secrets-store"
          readOnly: true
      env:                             # Or reference K8s secret
        - name: SECRET_KEY
          valueFrom:
            secretKeyRef:
              name: foosecret
              key: SECRET_KEY
Direct SDK / REST API Access

Applications use the Azure SDK or REST API to directly query Key Vault at runtime. Combined with Workload Identity, this is a clean, code-native approach that doesn't require volume mounts.

ℹ️
Best for apps that need to fetch secrets dynamically at runtime, or when you want to handle caching and versioning in application code.
Python — Azure SDK with Workload Identity 
from azure.identity import DefaultAzureCredential
from azure.keyvault.secrets import SecretClient
import os

# DefaultAzureCredential picks up Workload Identity token automatically
credential = DefaultAzureCredential()

vault_url = f"https://{ os.environ['KEYVAULT_NAME'] }.vault.azure.net"
client = SecretClient(vault_url=vault_url, credential=credential)

# Fetch secret — no password, no API key needed
secret = client.get_secret("my-database-password")
db_password = secret.value

# Fetch specific version
versioned = client.get_secret("my-secret", version="abc123...")
Node.js / TypeScript 
import { DefaultAzureCredential } from "@azure/identity";
import { SecretClient } from "@azure/keyvault-secrets";

const credential = new DefaultAzureCredential();
const client = new SecretClient(
  `https://${process.env.KEYVAULT_NAME}.vault.azure.net`,
  credential
);

// Fetch with retry and caching pattern
async function getSecret(name: string): Promise<string> {
  const { value } = await client.getSecret(name);
  return value!;
}
AAD Pod Identity (Legacy) Deprecated

The original Pod Identity solution for AKS. It allowed pods to assume Azure AD identities using node-level IMDS interception. Now officially deprecated in favor of Workload Identity.

⚠️
AAD Pod Identity is deprecated. Microsoft recommends migrating to Azure Workload Identity. No new features are being added and support will end. Do not use for new workloads.
Migration path from Pod Identity → Workload Identity
1. Enable OIDC issuer on existing cluster: az aks update --enable-oidc-issuer --enable-workload-identity
2. Create User-Assigned Managed Identity + federated credential
3. Assign Key Vault RBAC roles to the new identity
4. Update service account annotations
5. Remove AzureIdentity / AzureIdentityBinding CRDs
03 — Capabilities

Key Features

The AKS + Key Vault integration stack delivers a rich set of enterprise-grade capabilities for secrets management at scale.

🔄
Automatic Secret Rotation

The CSI driver polls Key Vault at a configurable interval (default 2 min). Updated secrets are reflected in mounted volumes automatically, without pod restarts for file-based access.

🆔
Workload Identity (OIDC)

Pods authenticate using projected service account tokens exchanged for Azure AD tokens. No stored credentials, client secrets, or certificates needed anywhere in the cluster.

🔑
Secrets, Keys & Certificates

Retrieve all three Key Vault object types. Certificates can be mounted as PEM or PFX, making TLS management seamless for ingress controllers and service meshes.

🔗
Kubernetes Secret Sync

SecretProviderClass can optionally sync KV secrets to native Kubernetes secrets, enabling compatibility with apps that read from env vars via secretKeyRef.

🔒
Private Endpoint Support

Key Vault can be locked down to a private endpoint within the AKS VNet, eliminating public internet exposure entirely for air-gapped or regulated environments.

📌
Version Pinning

Pin secrets to a specific version in SecretProviderClass for immutable deployments, or use empty string for always-latest — enabling controlled rollouts of credential rotations.

🌐
Multi-Vault Support

A single pod can mount secrets from multiple Key Vaults using multiple SecretProviderClass resources and volume definitions, supporting multi-tenant architectures.

📊
Audit & Diagnostics

All Key Vault access is logged in Azure Monitor and Microsoft Defender for Cloud. Full audit trail of who accessed what secret, when, and from which pod identity.

🛡️
Purge Protection & Soft Delete

Key Vault soft delete (90-day retention) and purge protection prevent accidental or malicious deletion of secrets — critical for compliance and disaster recovery.

04 — Implementation

Step-by-Step Setup Guide

Production-ready setup using the CSI Driver with Workload Identity — the recommended modern approach.

01
Enable OIDC Issuer & Workload Identity on AKS

Update your existing AKS cluster or create a new one with OIDC issuer and Workload Identity enabled. Also enable the CSI driver add-on.

Azure CLI 
# Enable on existing cluster
az aks update \
  --resource-group myResourceGroup \
  --name myAKSCluster \
  --enable-oidc-issuer \
  --enable-workload-identity \
  --enable-addons azure-keyvault-secrets-provider

# Or create new cluster with all flags
az aks create \
  --resource-group myResourceGroup \
  --name myAKSCluster \
  --enable-oidc-issuer \
  --enable-workload-identity \
  --enable-addons azure-keyvault-secrets-provider \
  --rotation-poll-interval 2m

# Get OIDC issuer URL (save this!)
export OIDC_ISSUER=$(az aks show \
  --name myAKSCluster \
  --resource-group myResourceGroup \
  --query "oidcIssuerProfile.issuerUrl" \
  --output tsv)
02
Create User-Assigned Managed Identity

Create a managed identity that your pods will assume. This identity will be granted Key Vault access.

Azure CLI 
# Create managed identity
az identity create \
  --name myIdentity \
  --resource-group myResourceGroup \
  --location eastus

# Export key values
export USER_ASSIGNED_CLIENT_ID=$(az identity show \
  --name myIdentity \
  --resource-group myResourceGroup \
  --query clientId -o tsv)

export USER_ASSIGNED_OBJECT_ID=$(az identity show \
  --name myIdentity \
  --resource-group myResourceGroup \
  --query principalId -o tsv)
03
Create Key Vault & Assign RBAC Roles

Create the vault (or use existing) and assign the "Key Vault Secrets User" role to the managed identity.

Azure CLI
# Create Key Vault with RBAC authorization az keyvault create \ --name myKeyVault \ --resource-group myResourceGroup \ --location eastus \ --enable-rbac-authorization true \ --soft-delete-retention-days 90 \ --enable-purge-protection true # Get KV resource ID export KV_ID=$(az keyvault show \ --name myKeyVault \ --query id -o tsv) # Assign 'Key Vault Secrets User' role az role assignment create \ --role "Key Vault Secrets User" \ --assignee-object-id $USER_ASSIGNED_OBJECT_ID \ --assignee-principal-type ServicePrincipal \ --scope $KV_ID # Add a test secret az keyvault secret set \ --vault-name myKeyVault \ --name "db-password" \ --value "SuperSecret123!"
04
Create Federated Credential & Kubernetes Service Account

Bind the Kubernetes service account to the Azure managed identity via an OIDC federated credential.

Azure CLI + kubectl 
# Create the service account in K8s
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: ServiceAccount
metadata:
  name: workload-identity-sa
  namespace: my-app
  annotations:
    azure.workload.identity/client-id: "$USER_ASSIGNED_CLIENT_ID"
EOF

# Create federated credential linking KSA → Azure Identity
az identity federated-credential create \
  --name myFederatedIdentity \
  --identity-name myIdentity \
  --resource-group myResourceGroup \
  --issuer "$OIDC_ISSUER" \
  --subject "system:serviceaccount:my-app:workload-identity-sa" \
  --audiences "api://AzureADTokenExchange"
05
Deploy SecretProviderClass & Pod

Create the SecretProviderClass manifest and deploy your application pod referencing it as a CSI volume.

Complete deployment YAML 
---
apiVersion: secrets-store.csi.x-k8s.io/v1
kind: SecretProviderClass
metadata:
  name: azure-kv-secrets
  namespace: my-app
spec:
  provider: azure
  secretObjects:
    - secretName: app-secrets
      type: Opaque
      data:
        - objectName: db-password
          key: DB_PASSWORD
  parameters:
    usePodIdentity: "false"
    clientID: "$USER_ASSIGNED_CLIENT_ID"
    keyvaultName: "myKeyVault"
    objects: |
      array:
        - |
          objectName: db-password
          objectType: secret
          objectVersion: ""
    tenantId: "$TENANT_ID"
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
  namespace: my-app
spec:
  template:
    metadata:
      labels:
        azure.workload.identity/use: "true"
    spec:
      serviceAccountName: workload-identity-sa
      volumes:
        - name: secrets-store
          csi:
            driver: secrets-store.csi.k8s.io
            readOnly: true
            volumeAttributes:
              secretProviderClass: "azure-kv-secrets"
      containers:
        - name: app
          image: myregistry.azurecr.io/myapp:latest
          volumeMounts:
            - name: secrets-store
              mountPath: "/mnt/secrets"
              readOnly: true
          env:
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: app-secrets
                  key: DB_PASSWORD
06
Verify & Monitor

Confirm the integration is working and set up monitoring for failed secret fetches.

kubectl — Verification Commands 
# Check CSI driver pods are running
kubectl get pods -n kube-system \
  -l app=secrets-store-csi-driver

# Check secret was mounted
kubectl exec -n my-app deploy/my-app -- \
  ls /mnt/secrets

# Read mounted secret file
kubectl exec -n my-app deploy/my-app -- \
  cat /mnt/secrets/db-password

# Check K8s secret was synced
kubectl get secret app-secrets -n my-app -o yaml

# Check SecretProviderClass status
kubectl describe SecretProviderClass azure-kv-secrets -n my-app

# View CSI driver logs for errors
kubectl logs -n kube-system \
  -l app=secrets-store-csi-driver --tail=50
05 — Security & Operations

Best Practices

Production-hardening checklist for AKS + Key Vault integrations in regulated and high-security environments.

  • ✓
    Use Workload Identity over Managed Identity Node-Level

    Workload Identity scopes authentication to individual pods (not entire nodes), preventing lateral movement if a pod is compromised. Always annotate service accounts and label pods with azure.workload.identity/use: "true".

  • ✓
    Apply Least-Privilege RBAC Roles

    Assign only Key Vault Secrets User (read-only) to application identities. Reserve Key Vault Secrets Officer for CI/CD pipelines and admin tooling. Scope roles at the secret level when possible, not vault level.

  • ✓
    Enable Purge Protection and Soft Delete

    Always enable both on Key Vaults used in production. Soft delete gives 90 days of recovery window; purge protection prevents immediate permanent deletion even by admins, which is required for compliance with standards like PCI-DSS and HIPAA.

  • ✓
    Use Private Endpoints for Production Vaults

    Attach a private endpoint in the AKS VNet and disable public network access on the vault. Combine with NSG rules and Azure Private DNS zones to ensure all traffic is internal and encrypted within the Azure backbone.

  • ✓
    Separate Vaults by Environment & Sensitivity

    Maintain distinct vaults per environment (dev/staging/prod) and optionally per sensitivity tier. This limits blast radius, simplifies RBAC management, and enables environment-specific rotation policies.

  • ✓
    Configure Secret Rotation Policies

    Define rotation policies on Key Vault secrets (e.g., rotate every 90 days) and leverage Event Grid or Azure Functions to trigger rotation workflows. Set the CSI driver rotation poll interval to 2 minutes for timely propagation.

  • ✓
    Avoid Syncing to K8s Secrets in High-Security Contexts

    Kubernetes secrets are stored as base64-encoded (not encrypted at rest by default) in etcd. If your etcd is not encrypted at rest, use file-based volume mounts instead of the secretObjects sync feature to avoid storing secrets in etcd.

  • ✓
    Enable Diagnostic Logs and Defender for Key Vault

    Route Key Vault audit logs to Log Analytics. Enable Microsoft Defender for Key Vault to get anomaly detection alerts (e.g., a pod suddenly accessing hundreds of secrets). Set up alerts for access denied events.

  • ✓
    Pin Secret Versions in Production Manifests

    For deployments where secret changes should be intentional and reviewed, pin the objectVersion in SecretProviderClass. Update versions via GitOps PRs to maintain an audit trail of credential changes.

  • ✓
    Use Azure Policy to Enforce Key Vault Integration

    Apply Azure Policy to require that all AKS deployments in specified namespaces use the CSI driver. Block the use of hardcoded environment variables for secrets via OPA Gatekeeper or Kyverno policies at the admission controller level.

06 — Summary

Key Takeaways

The most important principles to carry away from this guide when designing and operating AKS + Key Vault integrations.

01 / IDENTITY
Zero Credentials in Kubernetes

With Workload Identity, no client secrets, service principal passwords, or connection strings should exist anywhere in your cluster — not in ConfigMaps, not in environment variables, not in Helm values. The cluster is credential-free by design.

02 / DRIVER
CSI Driver is the Gold Standard

The Secrets Store CSI Driver with the Azure Key Vault provider is Microsoft's recommended approach. It requires no code changes, supports rotation, and works across all languages and runtimes through filesystem abstraction.

03 / ROTATION
Rotation is a First-Class Concern

Design your rotation strategy upfront. File-based mounts auto-refresh without restarts. K8s secret syncs update the secret object. Apps reading env vars will need a restart — consider using Reloader or similar controllers to handle this.

04 / NETWORK
Private Endpoints for Production

Public Key Vault endpoints are acceptable only for development. Any production, staging, or regulated workload must use private endpoints with public network access disabled. DNS resolution through Private DNS zones is mandatory.

05 / SCOPE
Identity Granularity Matters

Assign a separate managed identity per application team or microservice, not one shared identity for the whole cluster. This limits blast radius: if one workload is compromised, attackers cannot access secrets belonging to other applications.

06 / GITOPS
Treat SecretProviderClass as Code

Version-control your SecretProviderClass manifests alongside application code in Git. Secret version updates, new object additions, and RBAC changes should all flow through a reviewed pull request — creating an auditable, reproducible secrets configuration.
